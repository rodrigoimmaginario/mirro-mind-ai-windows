# v0.10.3 — Pi Startup Maintenance

**Date:** 2026-05-25

## Highlights

- Pi now uses `conversation-logger session-start --fast` during startup.
- Slow conversation maintenance runs in the background through
  `conversation-logger session-maintenance`.
- Maintenance output includes per-step timing for stale conversation closure,
  Pi session backfill, and pending extraction.
- The original blocking `session-start` path remains available for runtimes or
  manual use that want startup maintenance inline.

## Where We Started

Production validation after v0.10.2 showed that Pi could still spend a long time
opening Mirror. The status messaging improved, but the counter kept running and
the user still experienced startup as blocked. Manual timing confirmed the slow
path was conversation maintenance, not the release check.

The startup command had accumulated too much work: it unmutes logging, closes
stale conversations, backfills Pi JSONL sessions, and extracts pending
conversations. Extraction can call LLMs, so doing all of that before the runtime
feels open is the wrong boundary.

## What Changed

The conversation logger now has a fast startup path:

```bash
uv run python -m memory conversation-logger session-start --fast
```

This path only unmutes logging and returns immediately. Pi uses it during
`session_start` so the runtime can open and render the welcome without waiting
for historical maintenance.

The heavier work moved into:

```bash
uv run python -m memory conversation-logger session-maintenance
```

Pi launches this command in the background. The command reports per-step timing,
which makes future performance work concrete instead of speculative.

## Conscious Exclusions

This patch does not remove stale conversation closure, Pi session backfill, or
pending extraction. It changes when that work runs.

It also does not yet optimize the maintenance itself. If maintenance remains too
slow or too expensive in the background, the next step is to tune individual
stages, especially pending extraction.

## What We Learned

The product problem was not only slowness. It was blocked opening. Some
maintenance belongs to eventual consistency, not the critical path of entering
the runtime. A mirror can keep cultivating memory after the user has crossed the
threshold.

## Next Horizon

Production validation should confirm whether Pi's startup counter now stops
normally. If it still spins, the next investigation belongs to Pi lifecycle
completion rather than Python maintenance cost. The timing output will also show
whether pending extraction or backfill is the next optimization target.
