# v0.8.0 — Stable Self-Update Foundation

**Date:** 2026-05-22

## Highlights

- Runtime self-update now has a complete safe path: status, diagnosis, backup,
  dry-run, update check, fast-forward execution, migrations, and post-update
  health validation.
- The updater can repair itself through a code-only recovery lane when stale
  updater code crashes before normal update planning.
- Mirror Mind now distinguishes `main` as the integration/dogfooding channel and
  `stable` as the user-facing release channel.
- The welcome card shows installed version and update channel, so users can see
  what they are running before asking for help.
- Runtime release notes can be read through `python -m memory runtime
  release-notes` and natural-language Mirror requests such as "What's new in the
  latest Mirror Mind release?"

## Where We Started

Mirror Mind already had a strong local-first runtime, but update was still a
human memory problem. Production and development clones could be confused. A user
could not ask the runtime whether it was healthy enough to update. Backup existed
but was not part of a visible self-update route. Any push to `main` risked being
interpreted as a user-facing update, even though Ariad's release model says a
release is a closed, validated arc of work, not just a commit.

The first self-update slices made the runtime inspectable. Later slices made it
safe to mutate. The final pressure came from dogfooding: a stale updater crashed
while trying to install its own fix. That proved the updater needed a recovery
lane that did not require an end user to know git bootstrap commands.

## What Changed

The runtime now exposes a coherent operational surface:

- `runtime status` reports version, repository state, mirror home, database,
  migrations, extensions, clone role, update channel, Python version, and
  environment.
- `runtime diagnose` classifies drift and repair routes without mutating files or
  the database.
- `runtime backup` creates and structurally verifies database backups.
- `runtime version` shows the installed version, clone role, and update channel.
- `runtime update --check` checks the configured channel without mutating local
  refs.
- `runtime update --dry-run` plans from local refs without mutation.
- `runtime update` performs a backup-first, fast-forward-only update and applies
  migrations through a one-shot `MemoryClient` open.
- `runtime update --repair-updater` provides a code-only recovery lane when the
  full status gate cannot run because the updater itself is stale or broken.

The production/development boundary is now explicit through `.mirror-clone-role`.
The update target is explicit through `.mirror-update-channel`. Missing or invalid
update channels default to `stable`.

The welcome card now says what the runtime is:

```text
◇ Mirror · <user>
Version 0.8.0 · channel stable
...
```

Release notes are prospective from this release onward. Historical tags through
`v0.7.0` remain valid history, but `v0.8.0` is the first release that carries the
new release-management loop into the runtime itself.

## Conscious Exclusions

- This release does not complete CV9 or claim Mirror Mind 1.0 readiness.
- Release-aware update notices still fall back to commit counts when they cannot
  map an update to a release note; richer release-aware notices are planned next.
- Stable promotion is still a controlled manual process, not a fully automated
  command or Maestro workflow.
- Full natural-language release-note parity across every runtime surface remains
  follow-up work.
- The local production checkout may still have git branch `main` while following
  update channel `stable`; the channel, not the local branch name, controls the
  update target.

## What We Learned

Self-update is not one command. It is a chain of trust: know the current state,
refuse unsafe mutation, preserve recoverability, apply the smallest safe change,
explain what happened, and leave a future session able to understand the result.

The dogfooding incident where the updater could not install its own fix clarified
an important product rule: recovery paths must be usable by non-experts. Manual
git recovery is acceptable for development, but the product needs an updater
self-repair lane.

The release-channel split also clarified the relationship between Ariad and git.
`main` is where integration happens. `stable` is where a release becomes a user
promise.

## Next Horizon

The remaining Self-Update Done work is now clearer:

- make update notices and post-install output speak in release terms when a
  stable release note exists;
- ensure release-note access works naturally across supported runtime surfaces;
- add a repeatable release promotion checklist or doctor;
- provide a controlled promotion path for `stable`;
- validate a fresh clone updating from one stable release to the next without
  manual git intervention.
